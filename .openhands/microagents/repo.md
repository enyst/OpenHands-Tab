# Repository Guidelines


## Project structure overview

- Root
  - package.json: extension metadata, VS Code contributes/commands, scripts, deps
  - tsconfig.json / tsconfig.webview.json: TypeScript config for extension and webview
  - media/: compiled assets for webview (webview.js produced by build, webview.css in repo)
  - dist/: compiled extension JS (generated by build)
  - src/: TypeScript source for extension and webview
    - extension.ts: entry point, commands, webview setup, message bridge
    - connection/ConnectionManager.ts: HTTP+WS to agent-server; conversation management, reconnect, pause/resume
    - session/ConversationManager.ts: simple conversation_id storage helper
    - webview-src/: front-end TS and CSS for the webview (compiled into media/)
      - webview.ts: UI logic (messages, tool outputs, status)
      - webview.css: styles (source; also copied to media/)
  - tests/e2e/: E2E test scaffold using @vscode/test-electron + mocha
    - openTab.test.ts, diagnostics.test.ts: launch VS Code and run smoke scenarios
    - suite/index.js: test runner entry exporting run(); executes commands and logs diagnostics
    - AGENT_SERVER_QUICKSTART.md: how to run a compatible agent-server for deeper tests
  - .github/workflows/: CI workflows for build/package and E2E
  - .openhands/microagents/: environment helper docs

## Top-level docs in this directory
- README.md: overview, install, development flow, how to run with/without session API key
- README_DEV.md: quick development notes (build and launch)
- E2E_TESTING.md: manual, headless, and automated E2E testing guide (how to run locally/CI)
- prd.md: detailed product/architecture/design document, API mapping, requirements

## Commit messages

- Use real newlines in commit messages. Do not include literal \n sequences. For multi-paragraph messages, prefer one of:
  - git commit -m 'Subject' -m 'Body paragraph 1' -m 'Body paragraph 2'
  - git commit -F message.txt (where message.txt contains actual newlines)
  - git commit -m "Subject" && git commit --amend (to open editor and enter newlines)
- For feature branches and PRs, it is acceptable to rewrite history (reword/amend) before merge.
- Keep changes focused and avoid committing build artifacts or secrets.
