# agent-sdk-ts parity notes

## Context
- The TypeScript SDK docs describe a VS Code-focused implementation with local and remote conversation paths and a layered runtime/tooling stack.【F:docs/agent-sdk-architecture.md†L1-L159】
- The package-level guidance now reflects that `LocalConversation` runs an in-process orchestration loop against the VS Code workspace (no remote workspace/persistence).【F:packages/agent-sdk-ts/AGENTS.md†L13-L133】【F:packages/agent-sdk-ts/src/sdk/conversation/LocalConversation.ts†L22-L186】

## Gaps vs the Python agent-sdk
- **State persistence and recovery**: Python `LocalConversation` initializes `ConversationState` with an on-disk persistence directory and wires callbacks/visualizers before the loop runs, while the TypeScript version keeps an in-memory `EventLog` with no persistence or state hydration hooks.【F:openhands-sdk/openhands/sdk/conversation/impl/local_conversation.py†L104-L149】【F:packages/agent-sdk-ts/src/sdk/conversation/LocalConversation.ts†L37-L186】
- **Secrets and LLM registry**: Python maps provided secrets into the registry and subscribes LLMs to a registry/stats pipeline; the TS `SecretRegistry` currently reads from registered values, env vars, or VS Code SecretStorage when provided, and the LLM factory picks a single client instance without registry hooks or multi-model routing.【F:openhands-sdk/openhands/sdk/conversation/impl/local_conversation.py†L152-L163】【F:packages/agent-sdk-ts/src/sdk/runtime/SecretRegistry.ts†L12-L41】【F:packages/agent-sdk-ts/src/sdk/llm/factory.ts†L9-L82】
- **Workspace/tool negotiation for remote mode**: Python exposes remote and local workspaces with paged history fetching; the TS `RemoteConversation` always starts a `LocalWorkspace` session with a fixed tool list and does not let callers pass RemoteWorkspace-style options.【F:openhands-sdk/openhands/sdk/conversation/impl/remote_conversation.py†L12-L199】【F:packages/agent-sdk-ts/src/sdk/conversation/RemoteConversation.ts†L143-L184】

## Next steps to close parity and VS Code readiness
1. **Add persistence hooks to the TS runtime**: Introduce a persistence layer analogous to Python's `ConversationState.create(...)` so `EventLog` snapshots can be restored between VS Code sessions, and expose a persistence path in `LocalConversation` options.【F:openhands-sdk/openhands/sdk/conversation/impl/local_conversation.py†L104-L149】【F:packages/agent-sdk-ts/src/sdk/conversation/LocalConversation.ts†L37-L186】
2. **Broaden secrets + LLM flexibility**: Let the extension inject VS Code `SecretStorage` into `SecretRegistry` for scoped credentials, add per-conversation registration helpers, and build an LLM registry that can host multiple clients (OpenAI-compatible, Anthropic, Azure, future providers) with routing/usage reporting surfaced to the VS Code UI.【F:openhands-sdk/openhands/sdk/conversation/impl/local_conversation.py†L152-L163】【F:packages/agent-sdk-ts/src/sdk/runtime/SecretRegistry.ts†L12-L41】【F:packages/agent-sdk-ts/src/sdk/llm/factory.ts†L9-L82】
3. **Keep docs synced to VS Code scope**: Ensure the SDK docs clearly describe the in-process VS Code-only local loop, the lack of persistence today, and how secret storage/LLM registry injection should work once added so extension authors know the boundaries.【F:packages/agent-sdk-ts/AGENTS.md†L13-L133】【F:packages/agent-sdk-ts/src/sdk/conversation/LocalConversation.ts†L22-L186】
4. **Expand remote-conversation options**: Allow callers to specify remote workspaces/tooling and pass through the agent-server capabilities (e.g., MCP/skills) instead of the fixed local workspace + terminal/file/task trio, keeping history replay intact.【F:openhands-sdk/openhands/sdk/conversation/impl/remote_conversation.py†L12-L199】【F:packages/agent-sdk-ts/src/sdk/conversation/RemoteConversation.ts†L143-L184】
